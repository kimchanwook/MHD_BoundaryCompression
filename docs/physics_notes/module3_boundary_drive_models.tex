\docheader{Module 3: Boundary Drive Models}{Abstraction of the external drive from the MHD solver core}{Purpose: define the drive-model interface so that prescribed velocity, pressure loading, moving-wall logic, and future pulse variants can be compared within one solver architecture.}

\begin{overviewbox}
\textbf{What this note does.} This note separates the physical idea of the \emph{drive} from the numerical machinery of the solver. That separation is one of the most important architectural choices in the repository because it preserves extensibility without sacrificing clarity.
\end{overviewbox}

\symboltableheader
\texttt{driveModel} & drive-model structure & MATLAB structure containing the active drive type, parameters, and profile definitions.\\
\texttt{evaluate\_drive} & drive API & Function that returns the boundary prescription at the current time and location.\\
$v_n$ & normal velocity prescription & Inward or outward velocity applied normal to a wall.\\
$v_t$ & tangential velocity prescription & Velocity applied tangent to the wall, if used.\\
$p_{\text{bc}}$ & boundary pressure & Pressure value used by a pressure-loading model.\\
$s$ & boundary coordinate & Position coordinate measured along a wall.\\
$f_t(t)$ & temporal profile & Time envelope of the drive.\\
$f_s(s)$ & spatial profile & Variation of the drive along the wall.\\
$A$ & drive amplitude & Scalar strength parameter for the drive.\\
\symboltableend

\section{Why a drive abstraction is needed}
The core solver advances the ideal MHD equations. It should not have to know whether the boundary forcing is conceptually a wall moving inward, a pressure applied at the edge, or a pulse train with side-to-side time offsets. Those are different physical experiments, but they should all pass through a common interface.

Without an abstraction layer, every new drive concept would force edits throughout the code base. With an abstraction layer, the solver can simply ask:
\begin{quote}
At this boundary location and this time, what boundary state or prescription should be applied?
\end{quote}

That design keeps the project scientifically cleaner and also makes later optimization workflows possible.

\section{Separation of roles}
The repository should distinguish two layers clearly.

\subsection{Physical drive model}
This layer defines what the experimenter intends to do physically:
\begin{itemize}
    \item push inward with a given normal speed,
    \item apply pressure loading,
    \item move the effective wall,
    \item stagger wall timing,
    \item impose a spatial taper.
\end{itemize}

\subsection{Numerical boundary imposition}
This layer translates the physical intent into ghost-cell states or boundary fluxes. For example, a prescribed inward normal velocity is not itself a conserved variable. It must be converted into a boundary state consistent with the solver's primitive or conservative variables.

\begin{physicsbox}
\textbf{Architectural principle.} One should be able to change the \emph{physical drive family} without having to rewrite the conservative update, the flux routines, or the diagnostics. That is the main reason the interface exists.
\end{physicsbox}

\section{Baseline drive model: prescribed inward boundary velocity}
The first drive model is the simplest one that still captures the project's core question. The boundary is given a normal velocity prescription of the form
\[
v_n(s,t) = -A\,f_t(t)\,f_s(s),
\]
where the sign convention is chosen so that positive amplitude corresponds to inward compression.

The simplest baseline case chooses:
\begin{itemize}
    \item the same amplitude on every wall,
    \item the same temporal pulse on every wall,
    \item a spatially uniform profile along every wall.
\end{itemize}

This is the right first implementation because it is easy to interpret and easy to generalize.

\section{Pressure-loading concept}
A pressure-loading model would not prescribe velocity directly. Instead, it would specify a boundary pressure or pressure increment. Physically, that corresponds more closely to pushing on the plasma through stress rather than kinematic motion.

This is worth keeping as a future option because two drives can inject similar net impulse while producing different transient structures. A pressure-based drive may also be a more natural abstraction for some physical compression systems.

\section{Moving-wall concept}
A moving-wall model treats the boundary as a geometrically moving interface. That is a stronger statement than prescribed boundary velocity on a fixed grid. It moves the wall itself rather than only imposing a kinematic state at a stationary boundary location.

This model is more complicated numerically because one must decide whether the mesh moves, whether a mapped coordinate system is used, or whether the wall is represented through an immersed or cut-cell idea. That complexity is exactly why the project should begin with the fixed-grid prescribed-velocity model first.

\section{Pulse-train concept}
A pulse train applies repeated inward impulses rather than one single pulse. Physically, that may generate a sequence of inward-traveling fronts that interact constructively or destructively depending on timing.

This concept matters because repeated small pulses can sometimes create a more controlled accumulation of compression than one large impulsive event. The abstraction layer should therefore support a temporal profile that is either a single pulse or a repeated train of pulses.

\section{Asymmetric timing concept}
One of the scientifically useful perturbations is to break perfect wall synchrony. For example, one can delay the top boundary relative to the left, right, and bottom boundaries. This is a clean way to study how sensitive the compression quality is to timing mismatch.

The abstraction layer should support this by allowing each wall to carry its own temporal offset even when the underlying pulse shape is shared.

\section{Spatially varying drive concept}
The drive does not have to be uniform along a wall. It can be tapered, edge-weighted, center-weighted, or user-defined. That matters because local wall forcing can focus fronts differently and can either improve or worsen uniform compression in the interior.

A good interface therefore separates the temporal profile $f_t(t)$ from the spatial profile $f_s(s)$. That way one can compare, for example, a broad smooth pulse with uniform wall loading against the same pulse with a cosine taper at the corners.

\section{Recommended interface logic}
A useful conceptual interface is:
\[
\texttt{bcData = evaluate\_drive(driveModel, boundaryInfo, t, grid, stateInterior)}.
\]
The output \texttt{bcData} should indicate:
\begin{itemize}
    \item whether the boundary segment is active,
    \item which quantity is prescribed,
    \item what values should be imposed locally.
\end{itemize}

For example, the returned structure may contain fields such as:
\begin{itemize}
    \item \texttt{mode = 'prescribed\_normal\_velocity'},
    \item \texttt{vn},
    \item \texttt{vt},
    \item \texttt{p},
    \item \texttt{active}.
\end{itemize}

This pattern lets the boundary-condition routines remain generic.

\section{Relation between physical drive prescription and numerical boundary condition}
A common confusion is to treat the physical prescription as though it were already the numerical boundary state. It is not. For example, specifying a target normal velocity does not by itself determine the boundary density, pressure, tangential velocity, or magnetic field. One still must choose a boundary-state construction rule.

That is why the repository has separate directories for:
\begin{itemize}
    \item \texttt{src/drive\_models/} and
    \item \texttt{src/boundary/}.
\end{itemize}

The first decides what the drive wants. The second decides how that desire becomes a ghost-cell state compatible with the MHD solver.

\begin{keybox}
\textbf{Practical consequence.} The architecture is ready for future optimization only if the drive parameters are encapsulated cleanly. Once the drive is represented by a parameterized object and an evaluation function, one can later sweep or optimize amplitude, timing, width, taper, or asymmetry without altering the solver core.
\end{keybox}

\section{Concluding summary}
The drive-model abstraction is the architectural bridge between physics design and numerical implementation. The baseline model prescribes inward normal boundary velocity on a fixed Eulerian grid. Future models may instead impose pressure loading, moving-wall logic, repeated pulses, timing asymmetry, or spatial patterning. The key design rule is that these alternatives should enter through the same drive interface, while the ideal MHD solver core remains unchanged.
